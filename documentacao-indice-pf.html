1. Visão geral

O Índice PF é um painel que mede, semana a semana, quanto custa produzir uma porção de "prato feito" brasileiro (arroz, feijão, proteína, guarnição, salada e temperos). O preço de cada um dos 27 ingredientes é coletado automaticamente no Google Shopping, normalizado para uma unidade comum (R$/kg, R$/L, R$/bandeja etc.), e usado para calcular o custo de uma porção padrão. Os resultados são salvos no Supabase e exibidos em um dashboard Next.js.

3. Coleta de dados (scraping) — detalhado

Todo o processo de coleta vive em scraper_pf.py. Ele não faz scraping de HTML diretamente — usa a SerpAPI como intermediária, que já estrutura os resultados do Google Shopping em JSON. A chave da API está hardcoded no topo do arquivo (ver seção de segurança).

3.1 Catálogo de 27 ingredientes

Cada ingrediente é um dicionário Python com regras próprias de busca, validação e normalização, organizado em cinco categorias: Base, Proteínas, Guarnições, Salada e Temperos/Complementos. 

{
    "nome": "Frango (coxa/sobrecoxa)",
    "busca": "coxa sobrecoxa frango kg",      # termo enviado ao Google Shopping
    "unidade_base": "g",                       # unidade de normalização
    "palavras_ok":  ["frango", "coxa", "sobrecoxa"],   # pelo menos 1 deve aparecer no título
    "palavras_nao": ["temperado", "empanado", "nuggets", "processado", "defumado"],
    "qtd_porcao_g": 200,                        # gramas usados em 1 porção de PF
}
CampoFunção
buscaString literal enviada como query q para a SerpAPI
unidade_baseg, ml, unidade ou unid_peso_fixo — define como o preço é normalizado
palavras_okLista de termos — pelo menos um precisa estar no título do produto para ele ser aceito
palavras_naoLista de termos — se qualquer um aparecer no título, o produto é rejeitado (filtra variações erradas, ex: "feijão preto" quando se busca "feijão carioca")
qtd_porcao_gQuantidade (em gramas/ml) usada para calcular o custo de 1 porção de PF

3.2 Fluxo de busca por ingrediente

  1. Cache diário — antes de chamar a API, verifica se já existe uma resposta salva em cache_serpapi.json para aquele ingrediente na data de hoje. Se sim, reutiliza (evita gastar cota da SerpAPI em reexecuções no mesmo dia).
  2. Chamada à SerpAPI — requisição GET para https://serpapi.com/search com engine=google_shopping, gl=br, hl=pt, location=Brazil e o termo de busca do ingrediente. Retorna a lista de shopping_results.
  3. Limita aos 15 primeiros resultados de cada busca.
  4. Validação do produto (produto_valido) — aplica as regras palavras_ok / palavras_nao sobre o título em minúsculo. Produtos fora do escopo são descartados e contados como "rejeitados".
  5. Limpeza do preço (limpar_preco) — remove tudo que não é dígito, vírgula ou ponto do campo price da SerpAPI; trata formatos BR (1.234,56) e US (1234.56) antes de converter para float.
  6. Extração de link — tenta product_link, depois link, depois url; a SerpAPI raramente devolve link direto do produto no Google Shopping, então, se nenhum campo existir, monta um link de fallback para a busca no Google Shopping (clicável no modal de fontes do dashboard).
  7. Normalização por unidade — ver tabela abaixo. O objetivo é deixar todos os preços comparáveis (R$/g ou R$/ml), independente da embalagem anunciada.
  8. Filtro de outliers (IQR) (filtrar_outliers) — calcula Q1/Q3 dos preços normalizados e remove os que ficam fora de [Q1 − 1.5·IQR, Q3 + 1.5·IQR]. Só roda se houver 4 ou mais resultados válidos.
  9. Mediana (mediana) — calculada sobre os preços normalizados restantes; é o valor usado oficialmente como "preço do ingrediente" no resumo (a média/mín/máx também são guardados para exibição, mas calculados depois, em salvar_supabase.py).
  10. Cache e log — resultado salvo em cache_serpapi.json e impresso no console (válidos, rejeitados, outliers removidos).

3.3 Casos especiais de normalização por unidade

unidade_baseComo normalizaExemplo
g / ml (padrão) Extrai peso/volume do título via regex (extrair_quantidade: kg, litros, L, ml, g) e divide o preço por essa quantidade → R$/g ou R$/ml "Arroz 5kg R$ 24,90" → R$ 0,00498/g → exibido como R$ 4,98/kg
unidade (Ovo) Extrai nº de unidades do título (extrair_unidades: "30 unidades", "bandeja com X", "c/X" etc.) e converte para preço por bandeja de 30 "Ovos 20un R$ 15,00" → R$ 0,75/ovo → R$ 22,50/bdj30
unid_peso_fixo (Alface) Vendido por unidade no anúncio, mas sem peso declarado — usa um peso fixo estimado (peso_fixo_g, ex: 300g por pé) para converter em R$/g "Alface R$ 4,20/un" com peso fixo 300g → R$ 14,00/kg*
maco (Cheiro-verde, código existe mas nenhum ingrediente do catálogo atual usa) Extrai nº de maços do título (extrair_macos) e multiplica por um peso fixo por maço "Kit 2 maços" → divide preço por 2, depois pelo peso do maço

No dashboard, o label kg* aparece com um ícone de informação explicando que é uma estimativa baseada em peso fixo, e não em peso real anunciado pelo vendedor.

3.4 Cálculo do custo por porção e snapshot final

Na função main(), para cada ingrediente: calcula a mediana dos preços normalizados, converte para a unidade de exibição (R$/kg, R$/L, R$/bdj30...) e multiplica pela quantidade usada em 1 porção (qtd_porcao_g ou equivalente) para obter o custo_porcao. A soma de todos os custo_porcao é o custo estimado da porção completa do PF.

Tudo é então persistido em snapshot_pf.json, com esta estrutura:

{
  "data": "2026-06-17",
  "fonte": "Google Shopping via SerpAPI",
  "custo_total_pf": 23.45,
  "resumo": [ { "ingrediente": "...", "mediana_por_1000": ..., "label": "kg", "custo_porcao_r": ..., "qtd_resultados": ... } ],
  "resultados": [ { "ingrediente", "titulo", "preco_bruto", "preco_normalizado", "exibicao", "loja", "link" } ]
}

Este arquivo é o elo entre o scraper e o script que grava no banco (salvar_supabase.py).

4. Persistência no Supabase

salvar_supabase.py lê o snapshot_pf.json e grava em três tabelas via REST API do Supabase:

TabelaConteúdo
snapshots1 linha por data de coleta: data, fonte, custo total do PF
precos1 linha por ingrediente por snapshot: mediana, média, mínimo, máximo, desvio padrão, label, custo da porção, qtd de resultados coletados
resultados_brutosCada produto individual encontrado (título, loja, preço bruto, preço normalizado, link) — alimenta o modal "Fontes" do dashboard

Pontos relevantes do fluxo de gravação:

  • Se já existir um snapshot para a mesma data, reaproveita o id em vez de criar duplicado.
  • Estatísticas (calcular_stats) — média, mínimo, máximo e desvio padrão populacional — são recalculadas aqui a partir dos preco_normalizado agrupados por ingrediente (não vêm prontas do scraper).
  • Para precos e resultados_brutos, o script faz DELETE + INSERT (em vez de UPDATE/PATCH) antes de regravar — comentário no código indica que isso substituiu um PATCH que falhava com nomes de ingredientes acentuados.
  • Inserção dos resultados brutos é feita em lotes de 50 registros por chamada.

5. Geração de histórico simulado

gerar_historico.py existe para popular o dashboard com 8 semanas de histórico antes de haver dados reais suficientes. Ele parte de uma tabela fixa de preços-base (PRECOS_BASE) e de uma tendência semanal por ingrediente (TENDENCIAS, ex: alta de commodities, queda por safra) para "voltar no tempo" e gerar preços plausíveis para as 8 semanas anteriores à data de referência, com ruído aleatório de ±2% por semana (seed fixa random.seed(42) para reprodutibilidade). Os snapshots gerados são marcados com fonte: "simulado".

6. Limpeza dos snapshots simulados

limpar_simulados.py é pensado para rodar uma vez por semana, depois de cada coleta real. Ele busca o snapshot simulado mais antigo (campo simulado = true) e o remove (junto com seus preços e resultados brutos), de forma que, a cada execução semanal real, um snapshot falso é substituído por um real — até que, após 8 semanas, todo o histórico seja 100% real.

7. Automação (GitHub Actions)

O workflow .github/workflows/scraper-semanal.yml roda:

  • Agendamento: dias 1 e 15 às 11h UTC (8h em Brasília) — cron: '0 11 1,15 * *' = 2 coletas/mês (198 chamadas, dentro do plano gratuito SerpAPI de 250).
  • Disparo manual: via workflow_dispatch, pela aba Actions do GitHub
  • Passos: checkout → Python 3.11 + requests/python-dotenv → scraper_pf.py → salvar_supabase.py → calcular_custos_pratos.py. (O passo legado limpar_simulados.py foi removido — preservaria os dados antigos.)
  • Segredos usados: SUPABASE_URL, SUPABASE_KEY (service role), SERPAPI_KEY — todos do projeto novo.

8. Dashboard (frontend)

app/page.tsx é um componente client-side único que lê a tabela/view historico_precos do Supabase (via lib/supabase.ts, usando a anon key pública) e oferece duas abas:

8.1 Aba "Visão do Cliente"

  • Filtros: métrica de preço (mediana/média/mín/máx, com possibilidade de combinar várias), período por data, seleção de quais dos 27 ingredientes entram no cálculo.
  • Cálculo do custo do PF (calcularCustoPF): soma o custo de porção de cada ingrediente ativo; para proteínas, usa média ponderada entre as proteínas selecionadas (pesos diferentes para carne bovina/suína vs. frango/ovo), assumindo que o cliente serve uma proteína por porção, não todas somadas.
  • Gráfico 1: evolução do custo total do PF por semana, com banda de ±1 desvio padrão (real quando disponível na última coleta, estimada em ±8% para semanas anteriores que não têm desvio padrão calculado).
  • Gráfico 2: composição do custo por categoria (Proteína, Base, Guarnição, Salada, Temperos), alternável entre barras (% do total) e linhas (R$/porção).
  • Tabela detalhada: por ingrediente — mediana, média, mín, máx, desvio padrão, custo da porção e inflação no período, ordenável por coluna.
  • Modal de Fontes: ao clicar em "•••" numa linha, busca em resultados_brutos todos os produtos individuais coletados para aquele ingrediente/snapshot e mostra título, loja, preço e link (ou aviso de fallback quando a SerpAPI não retornou link direto).
  • Calculadora de custo: custo diário = custo da porção × nº de pratos/dia, com simulação de desconto no atacado configurável de 5% a 30%.

8.2 Aba "Backoffice" (admin)

Painel informativo (não executa nada): mostra os comandos do scraper, status do banco (nº de snapshots, registros, ingredientes) e a lista dos 27 ingredientes cadastrados com seu último preço conhecido.

9. Tabela canônica de ingredientes (em curadoria)

Etapa preparatória para expandir o scraping dos 27 ingredientes de um único PF para os 100 pratos regionais descritos em Tabela_Indice_PF_Ataulizada.xlsx (5 regiões, 925 linhas de ingrediente/quantidade). (A planilha tinha o prato 4 do Norte — Tacacá — grafado de duas formas, o que o partia em dois; unificado via PRATO_ALIAS em mapa_canonico.py.) O objetivo desta etapa é só consolidar uma base limpa de ingredientes — nada de scraping, banco ou mudança de cálculo ainda. Está pendente de revisão do responsável antes de avançar.

9.1 Problema de partida

A planilha Ingredientes_Scrapping.xlsx já havia deduplicado os ingredientes em 404 nomes canônicos, mas só resolveu a parte fácil: 394 dos 404 ainda eram texto de receita (ex: "Posta de Peixe (Cavala/Dourado)", "tiras de cenoura dura") e ~60 eram pratos preparados (Vatapá, Farofa, Pirão, Vinagrete), não produtos de varejo pesquisáveis no Google Shopping.

9.2 O que foi feito

Cada um dos 404 nomes foi classificado manualmente (julgamento caso a caso, não regex) em scripts/mapa_canonico.py como:

  • Atômico — fundido a um ingrediente-base de varejo (ex: as 16 variações de "Arroz branco" → um único item).
  • Composto — 60 preparos decompostos em ingredientes-base com proporções estimadas (ex: Farofa rica → farinha de mandioca + manteiga + ovo + cebola; Vatapá → pão + leite de coco + camarão seco + dendê + amendoim + cebola). Caldos/pirões tratam o líquido como subproduto sem custo.

9.3 Consolidação para o orçamento de API

A coleta usa a SerpAPI no plano gratuito: 250 chamadas/mês, 1 chamada por ingrediente por coleta. Para reduzir o nº de termos de busca distintos, uma camada de consolidação (CONSOLIDA em mapa_canonico.py) funde bases equivalentes seguindo regras definidas pelo responsável — cortes bovinos baratos (acém/músculo/agulha/moída) → um único "Acém bovino"; cortes nobres (alcatra/patinho/coxão/mignon) → "Alcatra bovina"; carne seca = charque = carne de sol; peixe de mar em posta → Pescada; peixe de rio → Pintado; feijão fradinho = feijão de corda; cebola única; entre outras — mais um lote de fusões equivalentes (manteiga de garrafa, queijos prato/mussarela, fubá/flocão, extrato/molho de tomate, pimenta, etc.).

Resultado: de 404 nomes brutos para 99 ingredientes-base (termos de busca distintos), o que permite 2 coletas completas por mês dentro do plano gratuito. Optou-se por preservar todos os ingredientes regionais (sem descartar itens de nicho), aceitando a cadência mensal menor em troca de dados completos.

O gerador scripts/gerar_tabela_canonica.py valida que os 404 estão 100% cobertos, que as proporções de cada composto somam 1 e que nenhuma base já fundida vaza para a lista final, depois produz tabela_canonica_ingredientes.xlsx com 8 abas: Receitas Finais (1021 linhas — prato × ingrediente canônico × porção em g, consolidado), Pratos (resumo) (100), Scraping (99 termos) (tripé, abaixo), Ingredientes Canônicos (99), Receitas Decompostas (925 reescritas, 1067 após decompor), Decomposições Aplicadas, Fusões Aplicadas e A Revisar (111 pontos de baixa confiança).

9.4 Verificações automáticas

  • Cobertura: 404/404 nomes mapeados (gerador falha se algum ficar de fora).
  • Conservação de massa: a soma das quantidades decompostas bate com a quantidade original em todas as 925 linhas.
  • Nenhum preparo composto sobrou na lista de ingredientes-base atômicos.
  • Todos os 100 pratos têm ao menos uma proteína; nenhum peso/contagem de ingredientes atípico (peso/porção entre 60g e 626g).

9.5 Proposta de tripé de scraping

Para cada um dos 99 ingredientes-base, scripts/tripe_scraping.py propõe o que o scraper precisa: termo de busca (query do Google Shopping), unidade (define a normalização do preço: 83 em g, 8 em ml, 1 em unidade = Ovo, e 7 em maco — folhas/verduras vendidas por maço/unidade sem peso no título, normalizadas por um peso_ref_g estimado, ex: alface 300g, cheiro-verde 150g) e as listas palavras_ok / palavras_nao, no mesmo estilo das 27 entradas originais. Ainda é proposta, pendente de revisão (aba Scraping (99 termos)).

Mudanças previstas no scraper_pf.py (não implementadas): (1) carregar o catálogo desse módulo em vez da lista hardcoded de 27; (2) deixar de calcular custo_porcao por ingrediente — no novo modelo o snapshot guarda só o preço normalizado por ingrediente-base, e o custo é calculado por prato (preço × quantidade da receita em Receitas Finais) numa etapa posterior. A pipeline de coleta (validação, limpeza de preço, normalização, filtro IQR, mediana) é reaproveitada sem mudança de lógica.

Decisões de design tomadas com o responsável: itens compostos são decompostos já nesta etapa; o modelo de custo futuro será por prato, não mais um índice único; preserva-se a completude (sem descartar itens regionais), aceitando ~2 coletas/mês no plano gratuito de API.

9.6 Banco: camada aditiva (preserva a estrutura atual)

A estrutura atual do Supabase é mantida — snapshots, precos, resultados_brutos e a view historico_precos seguem com as mesmas colunas; a expansão 27→99 é apenas mais linhas em precos/resultados_brutos. A lógica de coleta do scraper não muda. As únicas adições (em supabase_migration.sql, escrita defensiva com IF NOT EXISTS):

  • ingredientes (existia vazia) é recriada limpa (DROP + CREATE; 0 linhas, sem perda) com schema definido: nome, categoria, busca, unidade, peso_ref_g, palavras_ok/nao, ativo. Passa a ser o catálogo dos 99, referenciado por ingrediente_id (com FK religada).
  • Tabelas novas pratos (100), receitas (prato × ingrediente × qtd/porção, 1019 linhas) e custos_pratos (custo por prato por snapshot — calculado a cada coleta sem tocar em custo_total_pf/custo_porcao).

Continuidade de histórico: optou-se por recomeço limpo com os 99 nomes novos (o histórico antigo era majoritariamente simulado); a série real recomeça na próxima coleta. Pepino e Beterraba (dos 27 antigos) saem, pois nenhum dos 100 pratos os usa. O projeto Supabase foi migrado para uma instância nova; o gerador de histórico simulado (gerar_historico.py) foi removido.

9.7 Scraper reescrito (implementado)

O scraper_pf.py foi reescrito mantendo a lógica de coleta (SerpAPI, validação palavras_ok/nao, limpeza de preço, filtro IQR, mediana) e mudando três coisas:

  • Catálogo dinâmico: carrega os 99 ingredientes da tabela ingredientes (não mais lista hardcoded), com id para gravar ingrediente_id.
  • Normalização uniforme em R$/g: itens em g/ml extraem peso/volume do título; unidade (Ovo) e maco (folhas) usam contagem do título ÷ peso_ref_g — tudo vira R$/g, base do custo por prato (preço × quantidade da receita). Ex: bandeja de 30 ovos a R$22,50 → R$0,75/un ≈ R$15/kg.
  • Sem custo_porcao: o snapshot guarda só o preço normalizado por ingrediente; o custo é calculado por prato numa etapa separada.
  • Regras anti-inflação: palavras negativas globais (gourmet, premium, luxo, importado, cesta…) rejeitam produtos atípicos; e um filtro de sanidade descarta preços fora da faixa [mediana/4, mediana×4] do lote — pega erro de vírgula (R$40 → R$40.000) sem matar variação legítima entre lojas. Roda antes do filtro IQR já existente.

O salvar_supabase.py foi adaptado: grava ingrediente_id e mediana_normalizada (R$/g) em precos, sem custo_porcao (que vira cálculo por prato). Estrutura das tabelas inalterada.

A chave da SerpAPI deixou de ser hardcoded — passou a ser lida de SERPAPI_KEY (env / .env.local / secret do GitHub).

9.8 Preço fixo para itens não-cotáveis

Ingredientes sem produto de varejo equivalente (ex: sangue de cabidela/sarapatel) recebem unidade='fixo' e um custo_fixo em R$ (coluna adicionada via supabase_migration_2.sql). O scraper os ignora (não gasta SerpAPI) e o custo por prato usa o valor fixo. Hoje são 2 itens (Sangue suíno, Sangue de aves) a R$0,50 cada. Mecanismo reutilizável para outros itens não-cotáveis.

Além do custo fixo (flat), há o preço manual (preco_manual em R$/kg, via supabase_migration_4.sql): para ingredientes que não têm cotação online mas são parte relevante do prato (ex: jambu fresco R$60/kg, carne de bode R$40/kg). O scraper os ignora e o custo usa preco_manual × quantidade (escala com a porção, ao contrário do custo fixo). Total 111 ingredientes = 107 scrapeáveis + 2 fixos + 2 manuais. (Proteínas que dão nome ao prato — Filé mignon, Chuleta, Contrafilé, Patinho, Coxão mole, Carne moída, e os peixes Tambaqui/Pacu/Tainha/Cavala — foram des-consolidadas: têm cotação própria e nome fiel no prato, em vez de cair em Alcatra/Acém/Pintado/Pescada.)

O seed (scripts/seed_supabase.py, idempotente, requer service_role) popula ingredientes/pratos/receitas a partir da tabela canônica.

9.9 Custo por prato (implementado)

calcular_custos_pratos.py calcula, para um snapshot, o custo de cada um dos 100 pratos: custo = Σ (preço R$/g × qtd da receita) + Σ (custo fixo). Para ingrediente sem cotação na semana, usa a média dos últimos 3 valores em snapshots anteriores (fallback); sem histórico, o item não contribui (prato parcial). Grava em custos_pratos a cobertura: ingredientes_cobertos (cotação fresca + fixo), ingredientes_estimados (fallback) e ingredientes_total. Também define snapshots.custo_total_pf = mediana dos custos dos 100 pratos (índice nacional).

10. Observações de segurança

Chaves de API hardcoded no código-fonte. Durante a leitura do código, encontrei segredos commitados diretamente nos arquivos, e não apenas em variáveis de ambiente:

  • scraper_pf.py:8 — chave da SerpAPI hardcoded (SERP_API_KEY = "de201d8a..."). É uma chave paga; qualquer pessoa com acesso ao repositório pode usá-la.
  • gerar_historico.py:7-8 — URL e chave (JWT) do Supabase hardcoded como fallback, embora o resto do projeto use variáveis de ambiente (.env.local / GitHub Secrets).
  • lib/supabase.ts:3-4 e o fallback em salvar_supabase.py:15 usam a mesma chave anônima (role: anon) do Supabase — exposição esperada para uma anon key pública de frontend, desde que as políticas de RLS no banco restrinjam corretamente leitura/escrita.

Recomendação (não aplicada — apenas relatando): mover a chave da SerpAPI para variável de ambiente/secret e, por segurança, considerar rotacioná-la já que está exposta no histórico do git.
