2. Banco de dados e RLS

2.1 CRÍTICO — Escalação de privilégio: usuário pode se tornar admin.

Problema: a policy perfil_update (migração 7) permite UPDATE na própria linha de profiles sem restringir colunas. RLS controla LINHAS, não COLUNAS — então qualquer conta autenticada pode executar, via REST com a anon key pública, PATCH /rest/v1/profiles?id=eq.<uid> {"is_admin": true, "is_super": true}.

Por quê é grave: is_admin dá acesso a TODAS as contribuições (com localização dos usuários), CPFs/chaves PIX dos saques, aprovação de pagamentos, edição do índice publicado e gestão de anúncios. is_super dá exclusão de registros. É comprometimento total da aplicação a partir de qualquer cadastro grátis.

Solução (pronta — migração 33): privilégio de UPDATE por coluna: revogar o UPDATE genérico de authenticated e conceder só nas colunas de perfil (nome, telefone, região, sexo, nascimento, avatar, cpf, chave_pix, consentimento). As flags administrativas só mudam pelo SQL Editor/service role.

2.2 ALTO — Saque inserido direto na tabela, sem validação de saldo.

Problema: a policy pag_insert (migração 11) permite INSERT em pagamentos a qualquer autenticado (na própria linha). A validação de saldo/CPF/PIX era feita só no JavaScript do app — via REST dá para inserir um saque de qualquer valor.

Por quê: o pagamento é processado manualmente pelo admin, que pode não notar um valor acima do saldo real — risco financeiro direto e de poluição da fila de saques.

Solução (pronta — migração 33): remover o INSERT direto e mover a validação para o banco: RPC solicitar_saque(valor) security definer que confere saldo (contribuições aprovadas × valor por foto − reservado), CPF e PIX antes de inserir. O front já chama a RPC (com fallback até a migração rodar).

2.3 MÉDIO — Tabelas de operação legíveis por qualquer autenticado (conferir).

Problema: a aba admin lê precos_manuais_hist, login_log, audit_log e webhook_eventos com o client anon (RLS). Conferir no painel que cada uma dessas tabelas tem policy de SELECT restrita a eh_admin()/eh_super() — qualquer tabela criada sem policy explícita de select e com RLS desabilitado fica exposta via REST.

Por quê: vazariam trilhas de auditoria, IPs/dispositivos de login e histórico de preços manuais para qualquer conta.

Solução: auditoria única no SQL Editor: select tablename, rowsecurity from pg_tables where schemaname='public'; e revisar pg_policies. Ativar RLS em tudo; policies de admin onde faltar.

2.4 MÉDIO — Buckets de storage sem limite de tamanho/MIME.

Problema: os buckets contribuicoes e avatars aceitam upload de qualquer arquivo/tamanho (a policy só restringe a pasta ao uid). Um usuário pode subir arquivos enormes ou de tipos arbitrários.

Por quê: estouro de cota de storage (custo/negação de serviço) e distribuição de conteúdo arbitrário sob o domínio público do projeto (o bucket é público para leitura).

Solução: no painel Supabase → Storage, definir por bucket file size limit (ex.: 8 MB) e allowed MIME types (image/*).

2.5 BAIXO — Métricas de anúncio abertas a INSERT público.

Problema: anuncio_eventos aceita INSERT anônimo (por design — impressões/cliques de visitantes).

Por quê: um script pode inflar impressões/cliques, poluindo as métricas cobradas de anunciantes.

Solução: aceitável agora. Quando as métricas virarem cobrança: mover o registro para uma rota de API com rate limit por IP, ou deduplicar por sessão/dia no relatório.

2.6 BAIXO — Funções administrativas: postura correta (registro).

As RPCs sensíveis já seguem boa prática — security definer com set search_path, checagem eh_admin()/eh_super(), whitelist de tabelas no super_excluir (que preserva trilhas), e integrar_snapshot/aprovar_coletas_pendentes revogadas de anon/authenticated. Manter esse padrão em funções novas.

3. Rotas de API e webhooks

3.1 MÉDIO — Sem rate limiting em nenhuma rota.

Problema: /api/assinatura, /api/geo, /api/card e as mutações via Supabase (contribuições, eventos de anúncio) não têm limite de requisições.

Por quê: abuso barato — flood de criação de assinaturas no gateway, spam de contribuições (fotos), custo de banda/função.

Solução: Vercel WAF rate limiting nas rotas /api/* (painel da Vercel, sem código) e/ou checagem por usuário no banco (ex.: máx. N contribuições/dia — trigger simples).

3.2 MÉDIO — Webhook do Asaas: força do token é tudo.

Problema: a rota valida apenas o header asaas-access-token contra ASAAS_WEBHOOK_TOKEN. O Asaas não assina payloads; quem tiver o token ativa/cancela assinaturas.

Por quê: um token fraco/adivinhável permitiria marcar assinaturas como pagas (Premium grátis) ou cancelar assinaturas alheias.

Solução: ao configurar o gateway (pendência D9): gerar token aleatório de ≥32 bytes (openssl rand -hex 32), cadastrá-lo no Asaas e no Vercel env. A idempotência via webhook_eventos já está correta.

3.3 BAIXO — /api/assinatura: postura correta (registro).

Valida o access token do usuário com service role antes de agir, confere duplicidade de assinatura e não aceita valor do cliente (preço fixo no servidor). Manter.

4. Plataforma, headers e frontend

4.1 ALTO — Sem security headers.

Problema: next.config.ts está vazio — a aplicação responde sem Content-Security-Policy, X-Frame-Options/frame-ancestors, Strict-Transport-Security, Referrer-Policy, X-Content-Type-Options e Permissions-Policy.

Por quê: sem CSP, qualquer XSS que passe vira execução plena (com acesso ao token de sessão no localStorage — ver 4.3); sem frame-ancestors há clickjacking (embutir o site num iframe e roubar cliques de admin); sem HSTS, downgrade para HTTP em redes hostis.

Solução: bloco headers() no next.config.ts. Sugestão inicial (CSP em Report-Only primeiro, para calibrar sem quebrar):

Strict-Transport-Security: max-age=63072000; includeSubDomains
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(self), geolocation=(self), microphone=()
Content-Security-Policy: default-src 'self'; img-src 'self' data: blob:
  https://*.supabase.co; connect-src 'self' https://*.supabase.co
  https://nominatim.openstreetmap.org; style-src 'self' 'unsafe-inline';
  script-src 'self' 'unsafe-inline'; frame-ancestors 'none'

4.2 MÉDIO — Links e imagens de anúncios são conteúdo controlado por formulário.

Problema: anuncios.link e imagem_url viram href/src no site público. Hoje só admins cadastram, mas nada valida o esquema da URL — um javascript: em link executaria no clique.

Por quê: XSS armazenado no site público caso uma conta admin seja comprometida (ou pelo 2.1, antes do fix).

Solução: validar no cadastro (aba Anúncios) e/ou na renderização: aceitar apenas https://. Uma linha no AdCreative: ad.link?.startsWith('https://') ? ad.link : undefined.

4.3 MÉDIO — Token de sessão no localStorage (padrão supabase-js).

Problema: a sessão fica em localStorage, legível por qualquer script na página.

Por quê: amplia o impacto de um eventual XSS (roubo de sessão persistente). É o padrão da biblioteca e o app depende dele (usuarioDoStorage).

Solução: aceitar o padrão MAS reduzir a chance de XSS (CSP do 4.1, validação de URLs do 4.2). Alternativa futura: migrar auth para cookies httpOnly com @supabase/ssr — mudança grande, baixo retorno agora.

4.4 BAIXO — Lock de auth desabilitado no client.

Problema: lib/supabase.ts desativa o navigator.locks (workaround documentado de deadlock em navegação SPA/PWA).

Por quê: duas abas renovando o token ao mesmo tempo podem invalidar a sessão uma da outra (refresh token single-use) — é problema de disponibilidade, não de exposição.

Solução: manter (o retry de perfil já mitiga); reavaliar quando o supabase-js corrigir o deadlock upstream.

5. Dependências (npm audit de 09/07/2026 — 9 vulnerabilidades: 8 high, 1 moderate)

5.1 ALTO — Next.js com 4 CVEs na versão atual.

Problema: HTTP request smuggling em rewrites, DoS por buffering, bypass de checagem CSRF de Server Actions com origin null, crescimento ilimitado do cache de imagens.

Por quê: são CVEs publicados do framework que serve TODA a aplicação.

Solução: atualizar para o patch mais novo da linha 15 (npm install next@latest) e rodar o build/regressão. Baixo risco de quebra (mesma major).

5.2 MÉDIO — xlsx (SheetJS) sem fix disponível.

Problema: prototype pollution/ReDoS conhecidos, sem patch no npm. Usado só em lib/exportar.ts (exportação admin).

Por quê: o risco real é baixo (nós geramos o arquivo, não parseamos arquivos de terceiros), mas fica um alerta permanente no audit.

Solução: ou aceitar (documentado), ou trocar por exceljs, ou usar o build oficial do SheetJS via CDN deles (fora do npm). Recomendação: trocar por exportação CSV nativa (zero deps).

5.3 MÉDIO — react-simple-maps antigo puxa d3-color vulnerável (ReDoS).

Problema: a versão instalada depende de d3-color < 3.1.0.

Por quê: ReDoS explorável apenas se cores vierem de input do usuário — no app as cores são fixas; risco baixo na prática, mas é dívida.

Solução: subir react-simple-maps para a major nova (testar o MapaBrasil) ou substituir o mapa por SVG próprio (já é semi-estático).

5.4 BAIXO — ws (dependência transitiva do supabase realtime).

Solução: npm audit fix resolve sem breaking change. Incluir npm audit no checklist mensal.

6. Segredos e chaves

6.1 ALTO — Chave SerpAPI exposta em conversa.

Problema: a 4ª chave SerpAPI foi colada em texto plano numa conversa com IA (09/07/2026) antes de virar secret.

Por quê: logs/histórico de conversas são um canal de vazamento fora do seu controle; com a chave, terceiros esgotam sua cota (e coletas param de funcionar).

Solução: rotacionar no painel da SerpAPI e atualizar com gh secret set SERPAPI_KEY_4. Regra permanente: chave nova nasce direto no secret, nunca em chat.

6.2 BAIXO — Anon key e URL do Supabase hardcoded como fallback no repositório.

Problema: lib/supabase.ts e app/api/card/route.tsx trazem a anon key literal como fallback de env ausente (inclusive uma anon key ANTIGA no api/card, de antes da troca de projeto).

Por quê: a anon key é pública por design (vai ao browser de qualquer forma) — não é vazamento. O problema é operacional: o fallback mascara erro de configuração e a key antiga confunde. Além disso, o repositório é o lugar errado para qualquer credencial, por princípio.

Solução: remover os fallbacks (falhar alto se NEXT_PUBLIC_SUPABASE_* faltar) e apagar a key antiga do api/card. As envs já existem no Vercel.

6.3 BAIXO — Postura atual correta (registro).

.env* no gitignore (nunca commitado); service role só em GitHub Actions secrets e Vercel env (nunca no client); chaves SerpAPI em secrets. Manter.

7. Dados pessoais (LGPD)

7.1 MÉDIO — CPF e chave PIX em claro, duplicados em duas tabelas.

Problema: profiles.cpf/chave_pix e cópias em pagamentos. Legíveis por qualquer admin e presentes em backups em claro.

Por quê: CPF é dado pessoal sensível na prática (fraude); vazamento de banco expõe tudo, e a LGPD exige minimização e proteção proporcional.

Solução: curto prazo — mascarar o CPF na UI admin (só últimos 4 dígitos; o admin não precisa do CPF inteiro para conferir saque) e não copiar CPF para pagamentos (referenciar o perfil). Médio prazo — criptografar a coluna com pgcrypto (chave em env do servidor) e descriptografar só no fluxo de pagamento. Já existe consentimento registrado (consentimento_cpf_em) — bom.

7.2 MÉDIO — Localização precisa dos usuários nas contribuições.

Problema: lat/lng exatos de onde o usuário fotografou (casa/trabalho às vezes), visíveis a admins e usados no mapa de "Meus envios".

Por quê: padrão de deslocamento de uma pessoa é dado sensível; reter para sempre aumenta o passivo.

Solução: truncar coordenadas para 3 casas decimais (~110 m) no INSERT — precisão suficiente para validar a leitura por bairro/cidade — e política de retenção (ex.: apagar lat/lng após aprovação, mantendo cidade/UF).

7.3 BAIXO — Direitos do titular sem fluxo (exclusão de conta).

Problema: não há botão "excluir minha conta"; exclusão exige pedido manual.

Solução: fluxo de auto-exclusão (RPC que anonimiza profile + apaga fotos do storage) e página de política de privacidade (o link "Privacidade" do footer hoje não leva a lugar algum).

8. Operacional

8.1 MÉDIO — Sem 2FA nas contas de admin/super.

Problema: as contas com poder total (aprovar saques, editar índice) entram só com e-mail e senha.

Por quê: phishing/reuso de senha em outra base vazada compromete o app inteiro.

Solução: ativar MFA TOTP do Supabase e exigir para is_admin (checagem de aal2 nas policies de admin, ou no gate do /admin). Também: aumentar a senha mínima para 8+ e ligar a proteção contra senhas vazadas (painel Supabase → Auth).

8.2 MÉDIO — Backup e recuperação.

Problema: plano free do Supabase tem backup diário com retenção curta e sem PITR; nenhum backup externo dos dados (o índice histórico é o ativo do produto).

Por quê: um DELETE errado (ou conta comprometida — ver 2.1) pode destruir o histórico além da janela de backup.

Solução: job semanal (GitHub Actions) de pg_dump das tabelas de dados (snapshots, precos, custos_pratos, resultados_brutos, ingredientes, receitas) para artefato/storage privado.

8.3 BAIXO — Monitoramento de segurança.

Problema: as trilhas existem (audit_log, login_log, super_acoes — bom), mas ninguém é avisado de eventos anômalos (ex.: novo is_admin, rajada de saques).

Solução: trigger que registra mudanças de is_admin/is_super (já coberto pelo audit_log) + conferência quinzenal da aba Auditoria; futuramente webhook para notificação.

8.4 BAIXO — Projeto dentro do iCloud Drive.

Problema: o sync do iCloud já corrompeu refs do git e duplica arquivos de build ("arquivo 2.ts") — em 09/07 aconteceu 4 vezes.

Por quê: além do risco de corromper o repositório, o código (com eventuais segredos em .env.local) é sincronizado para a nuvem da Apple sem necessidade.

Solução: mover para fora do iCloud (ex.: ~/dev/indice-pf).
