1. Visão geral — o que é cada ferramenta

FerramentaPara que serveOnde vive
GTM — Google Tag Manager Um "container" de tags. Você instala um snippet no site e, dentro do painel do GTM, adiciona/edita tags (analytics, pixels, etc.) sem mexer no código nem fazer deploy. tagmanager.google.com
GA4 — Google Analytics 4 A ferramenta de analytics em si (visitantes, páginas, eventos, tempo real). Aqui o GA4 não vai no código do site — ele é disparado pelo GTM. analytics.google.com
GSC — Google Search Console Mostra como o site aparece na Busca do Google: quais páginas estão indexadas, por quais termos aparece, erros de rastreamento. Independente do GTM/GA4. search.google.com/search-console

Como se conectam nesta montagem:

Site (Next.js)  ──carrega──▶  GTM (container)  ──dispara──▶  GA4 (analytics)

Google Search Console  ──lê──▶  Site + sitemap.xml   (caminho separado)

A escolha aqui é medir o GA4 através do GTM: só o snippet do GTM entra no site. Assim, para adicionar qualquer tag futura (um pixel, um evento de clique) você mexe só no painel do GTM, sem tocar no código. É o modelo pedido para este projeto.

2. Ordem de configuração

  1. Criar a propriedade GA4 → anotar o Measurement ID G-XXXXXXX.
  2. Criar o container GTM → anotar o Container ID GTM-XXXXXXX.
  3. Colocar o GTM ID na variável de ambiente do site (o código já está pronto) e fazer deploy.
  4. Dentro do GTM, criar a tag do GA4 usando o G-XXXXXXX e testar no modo Preview.
  5. Publicar o container do GTM.
  6. Criar a propriedade no Search Console e verificar a posse do domínio.
  7. Submeter o sitemap no Search Console.
  8. Validar tudo (tempo real do GA4, Tag Assistant, indexação no GSC).

Faça na ordem. O GA4 vem antes do GTM porque você precisa do G-XXXXXXX para configurar a tag dentro do GTM. E publique o container só depois de testar no Preview — enquanto não publicar, nada é medido de verdade em produção.

3. Parte 1 — Criar a propriedade GA4

  1. Entre em analytics.google.com com sua conta Google.
  2. Administrar (engrenagem, canto inferior esquerdo) → coluna Conta → Criar → Conta. Nome da conta: Índice PF. Aceite os termos de compartilhamento de dados padrão.
  3. Em seguida, Criar propriedade. Nome: Índice PF. Fuso: (GMT-03:00) Brasília. Moeda: Real brasileiro (R$).
  4. Preencha "Detalhes da empresa" e objetivos (pode marcar "Gerar leads"/"Examinar o comportamento" — não muda a coleta).
  5. Na etapa Coletar dados, escolha a plataforma Web.
  6. Fluxo de dados Web:
    • URL do site: https://indicepratofeito.com.br
    • Nome do fluxo: Índice PF — Web
    • Deixe a Medição avançada (Enhanced Measurement) ligada — captura scroll, cliques em saída, buscas etc. automaticamente.
  7. Ao criar o fluxo, o Google mostra o ID de métricas (Measurement ID) no formato G-XXXXXXX. Copie e guarde. É o único dado do GA4 que você vai usar depois.

Não use o snippet de instalação ("Tag do Google") que o GA4 oferece nessa tela — nesta montagem o GA4 é carregado pelo GTM, não direto no site. Você só precisa do G-XXXXXXX.

4. Parte 2 — Criar o container do GTM

  1. Entre em tagmanager.google.com com a mesma conta Google.
  2. Criar conta. Nome da conta: Índice PF. País: Brasil.
  3. Container: nome indicepratofeito.com.br, plataforma Web. Criar e aceitar os termos.
  4. O GTM mostra o Container ID no formato GTM-XXXXXXX e um pop-up com dois trechos de código. Feche o pop-up sem copiar — a instalação no site já está feita por código (Parte 3). Só anote o GTM-XXXXXXX.

5. Parte 3 — Ligar o GTM no site

O código já foi aplicado neste repositório. Foram três mudanças:

  • Dependência @next/third-parties instalada (componente oficial do Next para GTM).
  • app/layout.tsx renderiza o GTM só se a variável NEXT_PUBLIC_GTM_ID existir.
  • O ID não fica hardcoded — vem da variável de ambiente.

Trecho relevante já presente em app/layout.tsx:

import { GoogleTagManager } from "@next/third-parties/google";

const gtmId = process.env.NEXT_PUBLIC_GTM_ID;

// ...dentro do RootLayout:
<html lang="pt-BR">
  {gtmId && <GoogleTagManager gtmId={gtmId} />}
  <body ...>

Falta só você definir a variável com o ID real. Passos:

  1. No Vercel (painel do projeto → Settings → Environment Variables):
    • Key: NEXT_PUBLIC_GTM_ID
    • Value: GTM-XXXXXXX (o ID da Parte 2)
    • Environments: marque Production (e Preview/Development se quiser testar antes).
  2. Redeploy na Vercel para a variável valer (ela é lida no build). Um novo push na branch de produção ou "Redeploy" no último deployment resolve.
  3. Local (opcional, para testar na sua máquina): adicione ao .env.local: 
    NEXT_PUBLIC_GTM_ID=GTM-XXXXXXX
    (O .env.local não é commitado — cada um define o seu.)

Por que o prefixo NEXT_PUBLIC_? No Next.js, só variáveis com esse prefixo ficam disponíveis no navegador. O GTM roda no navegador, então precisa ser público — e isto é normal: o Container ID do GTM não é segredo (fica visível no HTML de qualquer site que usa GTM).

Alternativa manual (não usada aqui, para referência): daria para injetar o GTM com next/script sem a dependência @next/third-parties. Como o componente oficial já cuida do script no <head> e do <noscript> de fallback, não há motivo para fazer manualmente.

6. Parte 4 — Configurar a tag do GA4 dentro do GTM

Agora o site carrega o GTM, mas o container ainda está vazio. Vamos dizer a ele para disparar o GA4.

  1. No GTM (tagmanager.google.com), abra o container indicepratofeito.com.br.
  2. Menu Tags → Novo.
  3. Clique em Configuração da tag → escolha Google Tag (nas versões mais novas do GTM esse é o tipo que substitui a antiga "GA4 Configuration").
  4. No campo ID da tag, cole o Measurement ID do GA4: G-XXXXXXX (da Parte 1).
  5. Clique em Acionamento (Triggering) → escolha All Pages (Todas as páginas — Page View). Isso faz a tag disparar em cada carregamento de página.
  6. Dê um nome à tag, ex.: GA4 — Google Tag, e clique em Salvar.

6.1 Testar no modo Preview (antes de publicar)

  1. No GTM, clique em Visualizar (Preview, canto superior direito).
  2. Na tela do Tag Assistant que abrir, digite https://indicepratofeito.com.br (ou a URL de Preview da Vercel, se testou lá primeiro) e conecte.
  3. O site abre numa aba com um selo "Tag Assistant Connected". Navegue por algumas páginas.
  4. De volta ao Tag Assistant, confirme que a tag GA4 — Google Tag aparece como Fired (disparada) nos eventos de página.
  5. Em paralelo, no GA4 → Relatórios → Tempo real, você deve se ver como 1 usuário ativo.

Se a tag não disparar: confirme que o deploy com a env NEXT_PUBLIC_GTM_ID já subiu (veja o código-fonte da página em produção — deve conter googletagmanager.com/gtm.js?id=GTM-XXXXXXX), e que o ID no site é o mesmo container que você está editando no GTM.

7. Parte 5 — Publicar o container

Enquanto o container não é publicado, a tag só existe no modo Preview — a medição real não acontece.

  1. No GTM, clique em Enviar (Submit, canto superior direito).
  2. Dê um nome à versão, ex.: v1 — GA4 via GTM, e uma descrição curta.
  3. Clique em Publicar.

A partir daqui, o GA4 passa a coletar dados de todos os visitantes em produção.

8. Parte 6 — Google Search Console

O GSC é independente do GTM/GA4. Ele precisa provar que você é dono do site. Há dois tipos de propriedade; recomendo a de domínio.

8.1 Recomendado — propriedade de Domínio (via DNS)

Cobre tudo de uma vez: http e https, www e sem www, e qualquer subdomínio. Requer acesso ao DNS do domínio.

  1. Em search.google.com/search-console → Adicionar propriedade → coluna Domínio → digite indicepratofeito.com.br.
  2. O Google mostra um registro TXT para adicionar no DNS, algo como google-site-verification=xxxxxxxxxxxx.
  3. Vá ao painel onde o DNS do domínio é gerenciado (onde você aponta o domínio para a Vercel — pode ser a própria Vercel, em Domains, ou o registrador onde comprou o .com.br, ex.: Registro.br). Adicione um registro:
    • Tipo: TXT
    • Nome/Host: @ (a raiz do domínio)
    • Valor: o google-site-verification=... que o GSC forneceu
  4. Salve e volte ao GSC → Verificar. A propagação do DNS pode levar de minutos a algumas horas; se falhar de primeira, espere e tente de novo.

8.2 Alternativa — propriedade de Prefixo de URL (via meta tag)

Se não tiver acesso ao DNS agora, dá para verificar por uma meta tag no HTML. O Next facilita isso pelo metadata — não apliquei esta opção no código para não commitar um token que ainda não existe. Se optar por ela:

  1. No GSC, escolha Prefixo do URL → https://indicepratofeito.com.br.
  2. Método de verificação Tag HTML. O Google dá um valor como <meta name="google-site-verification" content="ABC123..." />. Copie só o content.
  3. Em app/layout.tsx, dentro do objeto metadata já existente, acrescente: 
    export const metadata: Metadata = {
  metadataBase: new URL("https://indicepratofeito.com.br"),
  // ...campos já existentes...
  verification: { google: "ABC123..." },
};
  4. Faça deploy e clique em Verificar no GSC.

A verificação de domínio (8.1) é melhor a longo prazo. Prefira-a se conseguir editar o DNS; a meta tag (8.2) é o plano B.

9. Parte 7 — Sitemap e robots

Já foram criados no repositório, no padrão do App Router do Next 16:

  • app/sitemap.ts → gera https://indicepratofeito.com.br/sitemap.xml com as 4 páginas públicas indexáveis: /, /metodologia, /planos, /sobre. As áreas logadas (/admin, /perfil, /painel etc.) ficam de fora de propósito.
  • app/robots.ts → gera /robots.txt permitindo o rastreamento do site, bloqueando as áreas logadas e apontando para o sitemap.

Depois do deploy, confirme que abrem:

https://indicepratofeito.com.br/sitemap.xml
https://indicepratofeito.com.br/robots.txt

Submeter no GSC (após a propriedade estar verificada):

  1. GSC → menu lateral Sitemaps.
  2. Em "Adicionar novo sitemap", digite sitemap.xml e clique em Enviar.
  3. O status deve ficar Sucesso em pouco tempo (as páginas entram na fila de indexação — levar dias para aparecer na Busca é normal).

Quando novas páginas públicas forem criadas (ex.: um blog por prato, mencionado no roadmap), adicione as rotas na lista dentro de app/sitemap.ts. O sitemap não se atualiza sozinho com rotas novas.

11. Nota sobre LGPD / consentimento

O GA4 usa cookies para identificar visitantes. Sob a LGPD, o padrão de mercado é pedir consentimento antes de ativar analytics não essencial. Este guia não implementa banner de cookies nem o Consent Mode do Google — fica como decisão/etapa futura. Se for adotar depois, o caminho natural, já que tudo passa pelo GTM, é: subir um banner de consentimento e usar o Consent Mode v2 do Google Tag para só liberar a medição após o "aceitar". Como está montado (GA4 via GTM), dá para ligar isso sem tocar no código do site. Avalie com quem cuida do jurídico do projeto.
