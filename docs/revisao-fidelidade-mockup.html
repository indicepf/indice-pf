1. Contexto — por que esta revisão

A migração V0 → V1 está funcionalmente completa (Fases 1–10 do plano em docs/migracao-frontend-v1.html, PRs #64–#91). O CSS do mockup foi portado literalmente para app/globals.css (painéis, KPIs, segbar, filtros, region-bar, movers, tabelas, modais, auth, planos), mas o markup de várias seções foi montado a partir da estrutura extraída do código do mockup, sem comparação visual lado a lado. Resultado: o responsável apontou repetidamente divergências de posicionamento, composição de cabeçalhos e detalhes de estilo. Esta revisão fecha essa lacuna de uma vez, com verificação visual em cada passo — nada é dado como pronto sem screenshot comparado.

Regra da revisão: fidelidade absoluta ao mockup em estilo, posicionamento, espaçamento, cores e composição — com uma exceção deliberada: sem emojis (decisão do responsável em 08/07). Onde o mockup usa emoji como ícone, usar SVG equivalente ou omitir. Dados continuam 100% reais (nada das séries sintéticas ou números de marketing do mockup).

2. O que o responsável prepara antes

  1. Instalar/ativar a extensão Claude in Chrome (claude.ai/chrome), logado com a mesma conta do Claude Code, e abrir o Chrome antes de iniciar a sessão. É o item que muda o jogo: permite ao Claude abrir mockup e app lado a lado, tirar screenshots e comparar pixel a pixel, iterando até bater. Sem a extensão, o fallback é arrastar screenshots direto para o chat (imagens coladas como caminho /var/folders/… não são legíveis pelo Claude).
  2. Rodar as migrações 27 e 28 no SQL Editor do Supabase (se ainda não rodou a 28 na versão mais recente, rodar de novo — ela é idempotente e faz upgrade da tabela anuncios).
  3. Cadastrar 1 anúncio de teste por slot relevante em /admin → Anúncios (hero-lado, billboard, popup, gate-grafico) para os slots aparecerem na comparação visual.
  4. Estar logado como admin numa aba (para as telas destravadas) e ter uma aba anônima (para as telas públicas com lock).
  5. Opcional: listar por escrito qualquer divergência que já incomoda, por seção — entra como prioridade.

3. Método de trabalho da sessão

  1. Abrir o mockup (docs/indice-pf-ads2.html servido localmente) e a produção (indice-pf-ashen.vercel.app) em abas separadas do Chrome.
  2. Para cada seção do roteiro (§4): screenshot do mockup → screenshot do app → listar divergências (posicionamento, espaçamento, tamanhos, cores, tipografia, composição) → corrigir no código → re-screenshot até bater. Só então avançar.
  3. Um PR por bloco de seções (home, modais, páginas estáticas, auth, app logado), no fluxo padrão: branch → tsc/build → merge (deploy) → conferir em produção (atenção: o service worker do PWA segura a versão antiga no primeiro load — recarregar 2×).
  4. Nenhuma mudança de lógica de dados — só markup/estilo. Qualquer coisa funcional que aparecer vira nota, não desvio de rota.

5. Divergências já conhecidas (entrada da revisão)

  • Cabeçalhos dos gráficos — posicionamento dos filtros/controles não bate (apontado pelo responsável em 08/07; motivo: markup montado sem comparação visual).
  • Marca d'água do infinito no fundo do hero não foi portada.
  • Footer do mockup é escuro; o do app é claro.
  • Legenda clicável do gráfico geral (ligar/desligar séries) não existe no app.
  • KPIs: mockup mostra variação %; app mostra o índice em R$ por nível — precisa de decisão.
  • Modal de compartilhar: mockup tem abas "Redes sociais"/"E-mail completo" com previews.
  • Toggle "Comparar com período anterior" dos filtros do mockup não implementado.
