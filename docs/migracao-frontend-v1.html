1. Contexto e escopo

1.1 O que é a proposta

O arquivo indice-pf-ads2.html (334 KB, HTML único, auto-contido) é um protótipo de alta fidelidade do novo frontend, batizado "Índice PF by Infinity". Fatos verificados na leitura do arquivo:

  • 100% estático: zero requisições de rede, sem bibliotecas externas (sem React/CDN). Gráficos são SVG gerado por JS inline.
  • Dados sintéticos: os nomes de pratos (100), produtos (47) e regiões (5) estão embarcados em window.PF_EMBED, mas todos os preços e variações são gerados no navegador por um gerador pseudoaleatório determinístico (52 semanas de série falsa). Nenhum número de preço do mockup é real.
  • Estrutura: site público (home/dashboard, Metodologia, Planos, Sobre), login e cadastro em páginas dedicadas, app logado com sidebar (Meu painel, Enviar preços, Meus envios, Índice completo, Plano & assinatura, Checkout, Configurações).
  • Monetização: plano Premium a R$ 99,99/mês (checkout com Pix/cartão simulados), conteúdo bloqueado por assinatura (tabela de produtos por região, export, detalhe por produto) e 6 tipos de slot de publicidade (retângulo no hero, lateral, billboard, leaderboard, native in-feed e um "chart gate" — anúncio que cobre o gráfico até ser fechado).
  • Não cobre: mapa do Brasil (região vira botões/pílulas), telas de admin/moderação, backend, autenticação real, pagamento real, dark mode, rotas reais.

1.2 Decisões já tomadas

1.3 O que não pode quebrar

Tudo que está em produção continua funcionando durante e depois da migração: autenticação completa (login, cadastro, completar perfil com telefone validado, reset de senha, registrarLogin com GPS/dispositivo), contribuição de fotos (single e lote, compressão, dedup por hash, Nominatim, Storage), moderação no /admin, saques PIX, preços manuais, recálculo de custos, auditoria, trilhas de super, export CSV/XLSX, PWA e todas as políticas de RLS.

3. Princípios da migração

  • Fases sequenciais e mergeáveis. Cada fase = 1 ou mais PRs, cada PR deployável em produção sem quebrar nada. O app fica visualmente "híbrido" por pouco tempo, porque as Fases 1–2 já trocam a base visual de todas as telas.
  • Fluxo por PR (o mesmo já usado no projeto): branch → npx tsc --noEmit → npm run dev + verificação manual → PR → preview Vercel → merge em main (produção).
  • Lógica de dados não muda. lib/queries.ts (blend de preços, medianas, janelas, calibração) é reaproveitado integralmente; a migração adiciona 3 queries novas e não altera as existentes.
  • Convenção nova de organização: componentes compartilhados saem de app/ para components/ (pasta já existe, vazia): components/ui/, components/site/, components/shell/, components/dashboard/, components/ads/, components/premium/.
  • Aliases para transição de tokens: os nomes antigos (cream, paprika, line…) viram aliases dos valores novos na Fase 1, então nenhuma classe espalhada quebra; a renomeação acontece arquivo a arquivo conforme cada tela é tocada, e os aliases são removidos na Fase 11.
  • Sem números inventados. Percentuais, taxas de gateway e metas comerciais entram no documento apenas como pontos de decisão (seção 5).

5.0 — Níveis de preço extensíveis (pré-requisito, em lib/format.ts)

Substituir MODOS por uma estrutura que mescla os dois lados (decisão 3), preservando as keys atuais (compatíveis com o estado e com a calibração):

export type NivelPreco = {
  key: string; label: string;
  grupo: 'consumidor' | 'atacarejo' | 'industria';
  desconto: number | null;   // sobre o preço online
  nota: string; disponivel: boolean;
}
export const NIVEIS_PRECO: readonly NivelPreco[] = [
  { key: 'online',    label: 'Consumidor — online',  grupo: 'consumidor', desconto: 0,    nota: 'preço coletado no varejo online', disponivel: true },
  { key: 'mercado',   label: 'Consumidor — mercado', grupo: 'consumidor', desconto: 0.10, nota: 'estimativa −10% sobre o online',  disponivel: true },
  { key: 'atacarejo', label: 'Atacarejo',            grupo: 'atacarejo',  desconto: 0.22, nota: 'estimativa −22% sobre o online',  disponivel: true },
  { key: 'industria', label: 'Indústria',            grupo: 'industria',  desconto: null, nota: 'em breve',                        disponivel: false },
]
// alias de transição:
export const MODOS = NIVEIS_PRECO.filter(n => n.disponivel)

A UI renderiza disponivel: false como pill desabilitada "em breve". O percentual (ou fonte de dado) da Indústria é decisão pendente (D5) — sem ela o nível fica desabilitado, coerente com a decisão "só dados reais".

  • Tabela assinaturas: id, user_id → profiles, gateway, gateway_customer_id, gateway_subscription_id, status (pendente | ativa | inadimplente | cancelada), plano, valor, periodo_fim, criado_em, atualizado_em. RLS: usuário lê a própria (auth.uid() = user_id); escrita só via service role (nenhuma policy de insert/update para authenticated).
  • Função is_premium(uid uuid) returns boolean (SECURITY DEFINER, STABLE): existe assinatura status = 'ativa' com periodo_fim > now(). Usada dentro de policies e exposta como RPC.
  • Tabela webhook_eventos (gateway, evento_id unique, payload jsonb, processado_em) para idempotência de webhooks.

7.2 Gateway de pagamento — recomendação: Asaas (confirmar em D9)

Critério objetivo entre as opções brasileiras: o Asaas oferece assinatura recorrente nativa cobrando por Pix e cartão no mesmo objeto de assinatura; Stripe no Brasil aceita Pix só em cobrança avulsa (recorrência exige cartão); Mercado Pago (preapproval) centra a recorrência em cartão/saldo; Pagar.me orienta recorrência a cartão/boleto e a integração é mais voltada a plataformas. Checkout por link/QR gerado no servidor reduz a superfície de PCI no frontend. Taxas e prazos de repasse não estão neste documento de propósito — comparação comercial é decisão do responsável (D9). O desenho é agnóstico: trocar de gateway altera apenas lib/server/gateway.ts e a rota de webhook.

7.3 Server-side (padrão novo no projeto)

  • Env vars (Vercel Production + Preview; .env.local em dev): SUPABASE_SERVICE_ROLE_KEY, ASAAS_API_KEY, ASAAS_WEBHOOK_TOKEN, NEXT_PUBLIC_SITE_URL. As três primeiras nunca com prefixo NEXT_PUBLIC_.
  • lib/server/supabase-admin.ts NOVO: client com service role + import 'server-only' (adicionar o pacote server-only) — impede import acidental em componente de cliente.
  • lib/server/gateway.ts NOVO: wrapper do Asaas (criar cliente, criar/cancelar/consultar assinatura).
  • app/api/assinatura/route.ts NOVO (POST cria/cancela): como o app não tem SSR de auth, o cliente envia Authorization: Bearer <access_token> da sessão Supabase e a rota valida com supabaseAdmin.auth.getUser(token) antes de agir.
  • app/api/webhooks/asaas/route.ts NOVO: valida o header de token do webhook, grava em webhook_eventos (ignora duplicado), atualiza assinaturas (pagamento confirmado → ativa/estende periodo_fim; falha/estorno → inadimplente; cancelamento → cancelada) e registra em audit_log.

7.4 Gating — cliente E servidor

  • Cliente: estender app/useAuth.ts com isPremium (RPC is_premium ou select em assinaturas); a UI esconde/trava recursos.
  • Servidor (gating de verdade): os dados hoje são todos legíveis com a anon key via RLS pública — um overlay de CSS não protege nada. O que for premium-de-dados (tabela de produtos por região, export logado, detalhe por produto) passa a ser servido por view/tabela com policy is_premium(auth.uid()), ou por rota de API que verifica a assinatura. A lista exata do que é premium é decisão pendente (D10); o índice e os pratos permanecem públicos.

Risco: exposição de service role/API key se env vars forem mal configuradas; webhook processado em duplicidade sem a tabela de idempotência.

Verificação: sandbox do Asaas ponta a ponta no preview (criar assinatura de teste → webhook chega → is_premium vira true → recurso destrava); simular falha de pagamento; conferir no console do browser que a anon key não lê os dados premium.

6. Dependências e estratégia de rollout

F1 tokens ─► F2 componentes + re-skin ─► F3 navegação ─► F4 auth ─► F5 home ─► F6 shell logado
                                                                                    │
                                              F7 premium infra ◄────────────────────┘
                                                    │
                                              F8 checkout ─► F9 ads ─► F10 share ─► F11 polimento
  • F1–F2 são de risco baixo e entregam o visual novo cedo em todas as telas — evita meses de convivência de dois designs.
  • F3–F4 mexem em estrutura, não em lógica de dados.
  • F5 concentra o risco de regressão de números — mitigado pela comparação lado a lado com a home antiga no preview, mantendo a antiga em produção até o PR final da fase.
  • F7–F8 são aditivos: nada existente depende deles.
  • F9–F10 são plugáveis a qualquer momento após F5.
  • Branch/preview: uma branch por PR a partir de main; o preview do Vercel é o ambiente de validação. Atenção: o preview aponta para o MESMO Supabase de produção — testes de escrita sempre com contas de teste; para F7/F8, usar o sandbox do gateway com env vars distintas entre Preview e Production no Vercel.
  • Sem feature flags formais: a granularidade das fases + preview cobre a necessidade; a exceção é a F5 (home antiga preservada até o fim da fase).
